### F3 [P0] symbolic-coeff-even-power-poly-spurious-empty

- **Representante:** `solve(x^4+q=0, x)`
- **Correcto:** solve(x^4+q=0) = { ±(-q)^(1/4) } for q<=0 (→{0} at q=0), else No solution — never unconditionally empty. solve(x^4+q*x^2=0)=x^2(x^2+q): x=0 is a real root for every q, plus ±√(-q) for q<0.
- **Miembros (6):** `solve(x^4+q=0,x)`; `solve(x^4+q*x^2=0,x)`; `solve(x^6+q*x^2=0,x)`; `solve(x^4+q*x^3=0,x)`; `solve(x^6+q=0,x)`; `solve(x^8+q=0,x)`
- **Tractabilidad:** bounded-single-cycle
- **Causa-raíz:** UNIFIED ROOT CAUSE (both sub-mechanisms of the family reduce to one bug): `is_known_negative` in crates/cas_solver_core/src/isolation_utils.rs:46-61 conflates "not-provably-negative" with "provably-non-negative", so it wrongly reports a symbolic expression as KNOWN-negative. Two arms are unsound:
  - line 49  `Expr::Neg(_) => true`  — treats every syntactic negation as negative, but `Neg(e)` is negative only if `e` is provably POSITIVE. `-q`, `-1*q`, `-q^2` all become `Neg(...)` and are wrongly flagged negative.
  - line 50  `Expr::Mul(l,r) => is_known_negative(l) ^ is_known_negative(r)`  — the XOR only decides a product sign when BOTH factors have known sign; with a symbolic factor (e.g. `Neg(q)·x^2`) it treats the unknown factor as non-negative and concludes the product is negative.

Downstream, `classify_pow_base_isolation_route` (solve_outcome.rs:2782-2796) sees `exponent_is_even && rhs_is_known_negative` and picks `PowBaseIsolationRoute::EvenExponentNegativeRhsImpossible`, whose `even_power_negative_rhs_outcome(Eq)` (2822-2827) returns `SolutionSet::Empty`.
