# Engine Combination Ledger

This document is a derived strategy under
[ENGINE_IMPROVEMENT_AUTOMATION.md](/Users/javiergimenezmoya/developer/math/docs/ENGINE_IMPROVEMENT_AUTOMATION.md).

It should drive an iteration only when the ROI selector chooses `combination`
and there is already a causally coherent complementary hypothesis.

This document records engine changes that produced a strong local win but were
not retained because they regressed a global guardrail, usually
`embedded_equivalence_context`.

The purpose is not to preserve rejected patches as a backlog of random ideas.

The purpose is to preserve *combinable hypotheses*:

- local improvements that may become globally good when paired with
  - a cheap gate
  - a narrower call-site
  - cache or signature reuse
  - a follow-up patch that removes the new global cost they introduce

That makes this ledger part of the engine improvement workflow, not an archive
of failed work.

In practice the ledger has a second, equally load-bearing role: it is the
iteration journal of the auto-improvement loop. Retained cycles record their
capture, observations, decisions, and retained learnings here so later cycles
can reuse them.

Its roles are:

- preserve high-value local wins that failed global retention, explain why
  they failed, and make future combination attempts evidence-driven
- record retained iterations (`retained` / `retained-internal`) with their
  capture and retained learnings
- record observe-only discoveries with a reusable signature

Its role is not:

- to justify retrying every rejected optimization
- to override the benchmark guardrails
- to compete with `runtime`, `coverage`, or `observability` as a default mode

## When To Add An Entry

Add an entry when all of these are true:

- the local profiler or narrow slice improves materially
- the global guardrail gets worse enough that the change should not be retained
- there is a concrete technical hypothesis for why the local win did not scale
- there is a plausible complementary change that could make the idea safe later

Do not add entries for:

- trivial coding mistakes
- changes with no measurable local win
- changes whose only explanation is “benchmark noise”

## Required Fields

Entry headings are `## YYYY-MM-DD - title`. Every entry must carry:

- area:
  - orchestrator / calculus / integration / derive planner / cohesion / etc.
- status:
  - `retained`, `retained-internal`, `rejected`, `observe-only`,
    `discovery/observe-only`, `candidate for combination`, `superseded`
    (combined forms such as `retained-internal + discovery/observe-only`
    are allowed)
- observed:
  - what was measured or seen, with concrete evidence
- decision:
  - what was done and why
- retained learning:
  - the reusable lesson for future cycles

Optional named sections (use when they apply):

- capture: the auto-improvement capture fields (investment_class,
  cohesion_scope, calculus_matrix_cell, block_gate_check, ...)
- sibling sweep / adversarial verification / discovery (observe-only) /
  residual (observe-only)
- for combination-hypothesis entries (the original role): local lane,
  local win, global result, why it regressed globally, and what could make
  it combinable later

Older archived entries may use earlier schemas (candidate/files_changed/
investment); do not retrofit them.

## Rotation Policy

The active ledger keeps only recent months; older months are rotated into
`ENGINE_COMBINATION_LEDGER_ARCHIVE_YYYY_MM.md` files with
`scripts/engine_combination_ledger_tool.py --rotate YYYY-MM --reindex`.
Rotate past months when the active file exceeds roughly 2,000 lines.
Scorecard discovery metrics read the active ledger plus all archives, so
rotation never changes metrics. The entry index below is regenerated with
`--reindex` after adding entries; do not edit it by hand.

## Combination Rules

Combining rejected local wins is only worth doing when the combination is
causally coherent.

Good combination patterns:

- `expensive fast path + cheap gate`
- `repeated extraction + cache/reuse`
